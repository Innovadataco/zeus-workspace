# 8.27 CC-27: BKP (Respaldo y Recuperación)

## 8.27.1 Resumen Ejecutivo

El componente **CC-27 BKP** constituye el sistema de respaldo y recuperación de desastres (Disaster Recovery) del corredor vial Chía–Girardot, garantizando la continuidad operativa de todos los subsistemas ITS ante fallos críticos, incidentes de ciberseguridad, desastres naturales o errores humanos. Este subsistema habilita la protección integral de datos, configuraciones y estados operativos mediante estrategias de backup automatizadas, replicación geográfica y procedimientos de recuperación validados.

La capacidad de respaldo y recuperación representa una capa fundamental de resiliencia para la operación del Centro de Control y Operaciones (CCO), permitiendo la restauración de servicios críticos en tiempos definidos (RTO/RPO), la preservación de evidencia operativa y la minimización del impacto de incidentes disruptivos.

El subsistema integra tecnologías como snapshots instantáneos, replicación síncrona/asíncrona, deduplicación de datos y recuperación granular, garantizando la protección de la información en las condiciones operativas del corredor.

## 8.27.3 Descripción Funcional / Técnica

### Alcance de la Solución

El sistema BKP comprende una arquitectura integral de respaldo y recuperación que protege todos los componentes del ecosistema ITS: bases de datos operacionales, configuraciones de dispositivos de campo, imágenes de sistemas, registros de video, logs de eventos y documentación crítica. La solución implementa una estrategia de backup multinivel.

El subsistema opera de forma automatizada las 24 horas, ejecutando políticas de backup definidas por criticidad de datos: respaldos incrementales continuos para datos transaccionales, diferenciales cada 6 horas para configuraciones, y completos diarios para archivos estáticos. La replicación geográfica asegura que copias de seguridad cruciales existan fuera del sitio primario.

La arquitectura incluye capacidades de recuperación granular que permiten restaurar desde archivos individuales hasta sistemas completos, con validación automática de integridad mediante checksums criptográficos.

---

## 8.27.4 Justificación Estratégica

La implementación de un sistema robusto de respaldo y recuperación responde a la imperativa necesidad de garantizar la continuidad del servicio en un corredor de 306 km donde la interrupción prolongada de sistemas ITS puede tener consecuencias de seguridad vial, operativas y legales significativas.

La creciente sofisticación de amenazas cibernéticas, incluyendo ransomware y ataques dirigidos a infraestructura crítica, hace imprescindible contar con copias de seguridad inmutables y aisladas (air-gapped) que permitan recuperación limpia incluso ante compromiso total de sistemas primarios.

La protección de datos de video y transaccionales es especialmente crítica dado su valor como evidencia legal en incidentes viales. La retención garantizada y la recuperabilidad documentada de estos activos protegen tanto a la concesión como a los usuarios del corredor en procesos judiciales.

## 8.27.5 Especificación Técnica

| Parámetro | Especificación Mínima |
|-----------|----------------------|
| **Arquitectura de respaldo** | Estrategia 3-2-1: 3 copias, 2 medios diferentes, 1 offsite |
| **RTO (Recovery Time Objective)** | ≤ 4 horas para sistemas críticos; ≤ 24 horas para sistemas operativos |
| **RPO (Recovery Point Objective)** | ≤ 1 hora para datos transaccionales; ≤ 24 horas para datos operativos |
| **Frecuencia backup completo** | Diario para datos críticos, semanal para operativos |
| **Frecuencia backup incremental** | Cada 15 minutos para bases de datos transaccionales |
| **Retención datos operativos** | 90 días con acceso inmediato; 2 años en archivo |
| **Retención video/evidencia** | Mínimo 5 años con cadena de custodia |
| **Deduplicación de datos** | Relación mínima 5:1, global across all backups |
| **Compresión** | Algoritmos LZ4 o Zstd, ratio mínimo 2:1 |
| **Cifrado en reposo** | AES-256 para todos los respaldos |
| **Cifrado en tránsito** | TLS 1.3 para replicación entre sitios |
| **Inmutabilidad de backups** | Snapshots WORM (Write Once Read Many), retención mínima 30 días |
| **Replicación geográfica** | Síncrona para sitios < 100km; asíncrona para mayores distancias |
| **Validación de integridad** | Checksums SHA-256 verificados automáticamente post-backup |
| **Storage local edge** | Mínimo 7 días de retención en cada ubicación de campo |
| **Sitio DR secundario** | Infraestructura en ubicación geográfica diversa, mínimo 50km de separación |
| **Automatización** | Políticas programables, ejecución sin intervención manual |
| **Monitoreo** | Dashboard en tiempo real de estado de backups, alertas automáticas ante fallos |
| **Pruebas de recuperación** | Ejercicios trimestrales de restauración documentados y validados |

## 8.27.6 Requerimientos Funcionales (RF)

| ID | Nombre | Descripción Funcional / Técnica | Justificación Estratégica |
|----|--------|--------------------------------|---------------------------|
| **RF-01** | Backup Automatizado Programable | El sistema debe permitir definir políticas de backup programables por tipo de dato, criticidad y frecuencia. Debe soportar backups completos, incrementales y diferenciales con calendarización flexible y ejecución automática. | La automatización elimina errores humanos en la ejecución de respaldos y garantiza consistencia en la protección de datos. |
| **RF-02** | Replicación Geográfica entre Sitios | El sistema debe replicar copias de seguridad a sitio secundario remoto (≥ 50 km), con opciones de replicación síncrona (RPO cercano a cero) o asíncrona (balance costo/performance) según criticidad. | La replicación geográfica protege contra desastres locales (incendio, terremoto) que destruirían backups locales. |
| **RF-03** | Recuperación Granular de Datos | El sistema debe permitir restaurar desde archivos individuales hasta sistemas completos, con capacidad de buscar y recuperar versiones específicas de archivos por fecha. Debe soportar recuperación de bases de datos a punto en el tiempo (PITR). | La granularidad permite recuperar exactamente lo necesario sin restaurar sistemas completos, acelerando recuperación de errores menores. |
| **RF-04** | Backups Inmutables Anti-Ransomware | El sistema debe crear copias de seguridad inmutables (WORM) que no puedan ser modificadas ni eliminadas por ningún usuario, incluso administradores, durante período de retención configurado. | Los ataques ransomware buscan destruir backups; la inmutabilidad asegura siempre haya copia limpia disponible para recuperación. |
| **RF-05** | Validación Automática de Integridad | Cada backup debe ser verificado automáticamente mediante checksums criptográficos post-ejecución. El sistema debe generar alertas si se detecta corrupción y debe probar restauración periódicamente. | Los backups corruptos son tan inútiles como la ausencia de backup; la validación garantiza recuperabilidad real. |
| **RF-06** | Gestión de Retención y Compliance | El sistema debe aplicar políticas de retención automáticas según tipo de dato y normativa aplicable, moviendo datos entre tiers de almacenamiento y eliminando automáticamente al expirar retención. | El cumplimiento normativo requiere retención específica; la gestión automatizada evita riesgos legales y optimiza costos de almacenamiento. |
| **RF-07** | Recuperación de Desastres (DR) | El sistema debe incluir procedimientos documentados y automatizados para failover a sitio secundario, incluyendo cambio de DNS, activación de servicios, y validación post-recuperación. | La recuperación de desastres requiere coordinación de múltiples componentes; la automatización reduce RTO y errores humanos en momentos de crisis. |
| **RF-08** | Protección de Datos de Video/Evidencia | El sistema debe garantizar cadena de custodia digital para evidencia: timestamps certificados, hash criptográfico, registro de accesos, y almacenamiento conforme normativa de tránsito. | La evidencia de video sin cadena de custodia verificable no es admisible en procesos judiciales; la protección legal es crítica. |
| **RF-09** | Monitoreo y Alertas de Backup | El sistema debe proporcionar dashboard en tiempo real del estado de todos los backups, alertas inmediatas ante fallos, reportes de cumplimiento, y capacidad de forecast de capacidad de almacenamiento. | La visibilidad del estado de backups permite acción proactiva ante problemas antes de que ocurra una pérdida de datos. |
| **RF-10** | Orquestación de Recuperación | El sistema debe permitir orquestar recuperación de múltiples sistemas interdependientes en secuencia correcta (ej. base de datos antes que aplicación), con validación de cada paso antes de continuar. | Los sistemas ITS son interdependientes; la orquestación asegura recuperación consistente de todo el ecosistema. |

## 8.27.7 Requisitos No Funcionales (RNF)

| ID | Nombre | Descripción | Justificación |
|----|--------|-------------|---------------|
| **RNF-01** | Cumplimiento de RTO/RPO | Sistema debe garantizar RTO ≤ 4 horas para sistemas críticos y RPO ≤ 1 hora para datos transaccionales, medido y demostrado mediante ejercicios semestrales. | Los RTO/RPO definen la resiliencia operativa real; el cumplimiento debe ser verificable, no aspiracional. |
| **RNF-02** | Ciberseguridad de Backups | Backups cifrados AES-256, acceso basado en roles (RBAC), MFA para operaciones de recuperación, auditoría inmutable de accesos, aislamiento de red (air-gap) para copias críticas. | Los backups son objetivo prioritario de atacantes; deben ser tan seguros como los sistemas primarios. |
| **RNF-03** | Escalabilidad de Almacenamiento | Capacidad de escalar almacenamiento sin interrupción, soportar crecimiento de datos de 50% anual, y optimizar costos mediante tiering automático (hot/warm/cold). | El volumen de datos ITS (especialmente video) crece constantemente; el sistema debe acompañar ese crecimiento económicamente. |
| **RNF-04** | Rendimiento de Backup/Restore | Backups no deben impactar rendimiento de sistemas productivos (> 10% de degradación). Restauración de sistemas críticos en < 2 horas. | Los backups deben ser transparentes para operación; la recuperación debe ser rápida para minimizar impacto de incidentes. |
| **RNF-05** | Tolerancia a Fallos del Sistema de Backup | El sistema de backup mismo debe ser redundante, sin punto único de fallo, con replicación de metadatos y capacidad de recuperación ante falla del propio gestor de backups. | Un sistema de backup que puede fallar no es confiable; debe ser tan resistente como los sistemas que protege. |
| **RNF-06** | Compatibilidad Multi-Plataforma | Sistema debe soportar respaldo de sistemas heterogéneos: Windows, Linux, bases de datos SQL/NoSQL, VMs, contenedores, dispositivos de red, sistemas de archivos distribuidos. | El ecosistema ITS es tecnológicamente diverso; el backup debe ser unificador, no limitante. |
| **RNF-07** | Cumplimiento Normativo y Legal | Cumplimiento con regulaciones de protección de datos, retención de evidencia de tránsito, y estándares de la industria (ISO 27001, NIST Cybersecurity Framework). | El incumplimiento puede resultar en sanciones legales e invalidación de evidencia en procesos judiciales. |

## 8.27.11 Casos de Uso

### CU-01: Recuperación de Base de Datos tras Fallo de Hardware
**Actor Principal:** Administrador de Sistemas / Sistema BKP / TMS  
**Precondición:** Fallo de disco en servidor de base de datos principal  
**Flujo Principal:**
1. Se detecta fallo de RAID en servidor de producción, base de datos offline
2. Administrador inicia procedimiento de recuperación desde consola BKP
3. Selecciona punto de recuperación de 30 minutos antes del fallo (RPO cumplido)
4. Sistema BKP restaura imagen completa del servidor en hardware spare
5. Aplica logs de transacciones para recuperar hasta último commit
6. Base de datos disponible en 2.5 horas (RTO cumplido)
7. TMS reanuda operación normal sin pérdida de datos críticos

**Resultado:** Recuperación exitosa en RTO/RPO comprometidos, cero pérdida de datos de incidentes, continuidad operativa restaurada.

### CU-02: Recuperación ante Ataque de Ransomware
**Actor Principal:** Equipo de Seguridad / Sistema BKP / CCO  
**Precondición:** Sistemas comprometidos por ransomware, archivos cifrados  
**Flujo Principal:**
1. Se detecta ransomware en red, sistemas afectados aislados inmediatamente
2. Equipo de seguridad confirma que backups inmutables no fueron afectados
3. Se identifica punto de recuperación limpio de 6 horas antes del ataque
4. Se inicia recuperación masiva orquestada desde backups WORM
5. Sistemas restaurados en hardware limpio, verificados libres de malware
6. Servicios críticos restablecidos en 3.5 horas con datos íntegros
7. Investigación forense utiliza logs de backup para determinar vector de ataque

**Resultado:** Recuperación completa sin pago de rescate, lección aprendida: backups inmutables son la última línea de defensa.

### CU-03: Restauración de Evidencia de Video para Proceso Judicial
**Actor Principal:** Abogado / Administrador Legal / Sistema BKP  
**Precondición:** Requerimiento judicial de video de incidente de hace 18 meses  
**Flujo Principal:**
1. Abogado solicita video específico con fecha/hora/ubicación exacta
2. Administrador busca en sistema de archivo de backup histórico
3. Localiza backup de la fecha, verifica integridad mediante hash
4. Restaura video específico sin necesidad de recuperar todo el sistema
5. Genera cadena de custodia digital con timestamps certificados
6. Entrega evidencia al abogado con documentación de integridad
7. Video aceptado como evidencia en juicio por cumplir cadena de custodia

**Resultado:** Evidencia recuperada exitosamente después de 18 meses, cadena de custodia verificable, éxito en proceso judicial.

### CU-04: Failover a Sitio de Disaster Recovery
**Actor Principal:** Operador CCO / Sistema BKP / Equipo de DR  
**Precondición:** Incendio en edificio del CCO primario, evacuación total  
**Flujo Principal:**
1. Se declara desastre y se activa plan de DR
2. Equipo se traslada a sitio alterno ubicado a 80 km
3. Sistema BKP ejecuta failover automatizado: activa réplicas en sitio DR
4. Cambia DNS para apuntar a servicios en sitio alterno
5. Operadores inician sesión en sistemas restaurados en sitio DR
6. Operación del corredor continúa con datos de hasta 15 minutos antes del incidente
7. CCO primario restaurado y se ejecuta failback una vez recuperado

**Resultado:** Continuidad operativa mantenida durante desastre físico, RPO de 15 minutos y RTO de 3 horas cumplidos.

### CU-05: Recuperación de Configuración tras Error Humano
**Actor Principal:** Ingeniero de Red / Sistema BKP / Dispositivos de Campo  
**Precondición:** Configuración incorrecta masiva aplicada por error a 50 dispositivos  
**Flujo Principal:**
1. Ingeniero detecta que cambio masivo de configuración causó fallas
2. Necesita revertir configuraciones a estado de 24 horas antes
3. Accede a sistema BKP y busca backups de configuración por dispositivo
4. Selecciona punto de recuperación y dispositivos afectados
5. Sistema restaura configuraciones automáticamente vía red de gestión
6. Verifica que dispositivos vuelven a operar normalmente
7. Documenta lección aprendida sobre procesos de cambio

**Resultado:** Recuperación rápida de error humano, 50 dispositivos restaurados en 45 minutos sin visitas de campo.
