# 8.1 CC-18: BCS/VRU (Bicycle Counter / Vulnerable Road User)

## 8.1.1 Resumen Ejecutivo

El componente **CC-18 BCS/VRU** constituye el sistema de conteo y monitoreo de usuarios vulnerables de la vía (Vulnerable Road Users) del corredor Chía–Girardot, específicamente diseñado para detectar, contar y monitorear el flujo de ciclistas, peatones y otros usuarios no motorizados en la infraestructura vial. Este subsistema es fundamental para la movilidad sostenible y la seguridad de usuarios vulnerables.

El monitoreo de usuarios vulnerables representa un componente emergente de sistemas ITS modernos, reconociendo que la infraestructura vial es compartida por múltiples modalidades de transporte. El sistema BCS/VRU permite cuantificar el uso de la infraestructura por ciclistas, identificar patrones de demanda, y mejorar la seguridad mediante detección de conflictos entre vehículos motorizados y no motorizados.

El componente BCS/VRU se despliega en corredores ciclistas, puentes compartidos, zonas peatonales adyacentes a la vía, y puntos de intersección crítica donde ciclistas y vehículos comparten el espacio. Los datos recopilados apoyan la planificación de infraestructura ciclista y la evaluación de intervenciones de seguridad.

## 8.1.3 Descripción Funcional / Técnica

### Alcance de la Solución

El sistema BCS/VRU comprende múltiples tecnologías de detección integradas para capturar el movimiento de usuarios vulnerables:

- **Conteo de ciclistas (BCS)**: Detección y conteo automatizado en corredores ciclistas dedicados o compartidos
- **Detección de peatones**: Monitoreo de flujo peatonal en cruces, puentes peatonales y zonas de acceso
- **Clasificación de usuarios**: Diferenciación entre ciclistas, peatones, patinetas, sillas de ruedas, etc.
- **Monitoreo de conflictos**: Detección de situaciones de riesgo entre usuarios vulnerables y vehículos motorizados

Tecnologías de detección incluyen:
- Sensores inductivos (bucles) en ciclocarriles
- Cámaras con analítica de video (AI) para detección y clasificación
- Sensores infrarrojos pasivos (PIR) y activos
- Radares de baja potencia optimizados para VRU
- Sensores acústicos para detección de bicicletas

El sistema opera de forma continua, recolectando datos estadísticos de flujo, velocidad, dirección y patrones temporales. La información se transmite al CCO para visualización en dashboard y se integra con sistemas de información al usuario (mostrando conteos en paneles informativos).

### Justificación Estratégica

La movilidad en bicicleta y a pie es prioridad en políticas de sostenibilidad urbana:
- **Cero emisiones**: No contribuyen a contaminación atmosférica
- **Salud pública**: Promoción de actividad física
- **Equidad**: Modo de transporte accesible económicamente
- **Eficiencia**: Uso eficiente del espacio vial

Sin embargo, los usuarios vulnerables enfrentan riesgos elevados:
- Mayor exposición en accidentes con vehículos motorizados
- Invisibilidad en estadísticas de tráfico tradicionales
- Infraestructura frecuentemente insuficiente o insegura

El sistema BCS/VRU proporciona:
- **Datos para planificación**: Justificar inversión en infraestructura ciclista
- **Seguridad**: Detectar conflictos y activar alertas
- **Información**: Mostrar datos de uso que incentivan más usuarios
- **Evaluación**: Medir impacto de intervenciones de seguridad

## 8.1.4 Especificación Técnica

| Parámetro | Especificación Mínima |
|-----------|----------------------|
| **Tecnologías de detección** | Video AI + sensores inductivos + radar (redundancia) |
| **Precisión de conteo** | ≥ 95% para ciclistas, ≥ 90% para peatones |
| **Velocidad de detección** | 5-40 km/h para ciclistas, 0-10 km/h para peatones |
| **Capacidad de clasificación** | ≥ 4 clases: ciclista, peatón, patineta/scooter, otros |
| **Ancho de detección** | Configurable: 1-6 m según ubicación |
| **Procesamiento** | Edge computing, análisis en tiempo real < 100 ms |
| **Comunicación** | Ethernet IP, 4G/5G, LoRaWAN para ubicaciones remotas |
| **Almacenamiento local** | ≥ 30 días de datos agregados |
| **Energía** | 12-24 VDC, opción solar con baterías |
| **Protección ambiental** | IP65 para electrónica, IP67 para sensores de campo |
| **Rango térmico** | -10°C a +50°C operación, -20°C a +60°C almacenamiento |
| **Privacidad** | Procesamiento local, no almacenamiento de video/imágenes |
| **Resolución temporal** | Datos agregados por 15 minutos, eventos en tiempo real |
| **Certificaciones** | CE, FCC, cumplimiento GDPR/privacidad |

## 8.1.5 Requerimientos Funcionales (RF)

| ID | Nombre | Descripción Funcional / Técnica | Justificación Estratégica |
|----|--------|--------------------------------|---------------------------|
| **RF-01** | Conteo Automático de Ciclistas | El sistema debe detectar y contar automáticamente ciclistas en corredores ciclistas o carriles compartidos, diferenciando dirección de flujo. La precisión debe ser ≥ 95% en condiciones normales (día, clima despejado) y ≥ 85% en condiciones adversas (lluvia, oscuridad). | El conteo preciso es fundamental para justificar inversiones en infraestructura ciclista. Los datos de volumen sustentan decisiones de ampliación de corredores y conexiones. |
| **RF-02** | Detección y Conteo de Peatones | El sistema debe detectar y contar peatones en pasos peatonales, puentes peatonales, y zonas de cruce críticas. Debe diferenciar entre peatón individual y grupos. La precisión debe ser ≥ 90% para individuos, ≥ 85% para grupos. | Los peatones son los usuarios más vulnerables. El monitoreo permite identificar puntos de alta demanda peatonal donde se requiere mayor seguridad. |
| **RF-03** | Clasificación de Tipo de Usuario | El sistema debe clasificar usuarios detectados en al menos 4 categorías: ciclista, peatón, patineta/eléctrico, y otros (silla de ruedas, carreola, etc.). La clasificación debe reportarse con nivel de confianza. | Diferentes tipos de usuarios vulnerables tienen necesidades distintas de infraestructura. La clasificación permite análisis específico por modalidad. |
| **RF-04** | Detección de Conflictos VRU-Vehículo | El sistema debe detectar situaciones de riesgo donde un usuario vulnerable y un vehículo motorizado compartan espacio de manera insegura: ciclista en carril vehicular sin ciclocarril, peatón en calzada, acercamiento peligroso. Debe generar alerta al CCO. | La detección temprana de conflictos permite intervención preventiva: activar señalización de advertencia, alertar a patrullas cercanas, o documentar zonas de alto riesgo para mejoras. |
| **RF-05** | Datos Estadísticos y Agregados | El sistema debe generar reportes automáticos con: volumen por hora/día/semana/mes, patrones temporales (horas pico), distribución direccional, velocidades promedio (ciclistas), y clasificación de usuarios. Los datos deben exportarse en formatos estándar (CSV, JSON, XML). | Los datos agregados sustentan estudios de factibilidad, evaluaciones de impacto, y reportes de sostenibilidad. La exportación estándar facilita integración con otras plataformas. |
| **RF-06** | Visualización en Tiempo Real | El CCO debe poder visualizar en dashboard: conteos actuales, tendencias del día, mapa de calor de actividad, y alertas de conflictos activos. Debe permitir consulta histórica por ubicación y período. | La visualización en tiempo real permite identificar anomalías (eventos que atraen ciclismo masivo) y coordinar respuestas operativas (cierres temporales, desvíos). |
| **RF-07** | Integración con DMS/VMS | El sistema debe poder enviar datos de conteo a paneles de mensajería variable (CC-03) para mostrar información tipo "1,234 CICLISTAS HOY" o "RUTA CICLISTA CONGESTIONADA". La integración debe ser configurable por ubicación. | La retroalimentación pública de datos de uso incentiva la adopción de ciclismo (efecto "otros lo hacen") y proporciona información útil para usuarios planificando su ruta. |
| **RF-08** | Privacidad por Diseño | El sistema debe procesar todo el análisis de video localmente en edge, sin transmitir ni almacenar imágenes o video al CCO. Solo datos agregados y anónimos deben transmitirse. No debe ser posible identificar individuos específicos. | La privacidad de usuarios es obligatoria legalmente (Ley 1581 de 2012). El monitoreo de VRU no debe convertirse en vigilancia de individuos. El procesamiento local garantiza anonimato. |
| **RF-09** | Calibración Automática | El sistema debe incluir calibración automática que se adapte a cambios de iluminación, clima, y patrones de uso sin intervención humana. Debe detectar cuando su precisión se degrada y solicitar mantenimiento. | La calibración manual frecuente es costosa en corredores extensos. La autocalibración reduce costos operativos y mantiene precisión a lo largo del tiempo. |
| **RF-10** | Detección en Condiciones Adversas | El sistema debe operar en condiciones de lluvia, niebla, oscuridad, y temperaturas extremas con degradación controlada de precisión (< 10% vs. condiciones normales). Debe incluir sensores redundantes (video + inductivo) para mantener operación ante fallo parcial. | El clima en el corredor Chía-Girardot varía significativamente. El sistema debe ser confiable año completo, no solo en condiciones ideales. |

## 8.1.6 Requisitos No Funcionales (RNF)

| ID | Nombre | Descripción | Justificación |
|----|--------|-------------|---------------|
| **RNF-01** | Disponibilidad del Sistema | El sistema BCS/VRU debe mantener disponibilidad ≥ 98% para conteo estadístico, medida mensualmente. La detección de conflictos debe tener disponibilidad ≥ 99%. | Aunque no es sistema de seguridad crítica como CCTV, la disponibilidad afecta la calidad de datos para planificación. Los períodos de fallo crean huecos en series temporales. |
| **RNF-02** | Precisión y Validación | El sistema debe incluir mecanismos de autovalidación que comparen conteos con métodos alternativos periódicamente. Debe reportar su margen de error estimado con cada dato. | Los datos inexactos pueden llevar a decisiones erróneas. La transparencia sobre precisión permite que usuarios de datos ponderen adecuadamente la información. |
| **RNF-03** | Latencia de Detección | El tiempo desde que un usuario pasa el punto de detección hasta que el conteo se refleja en el sistema debe ser < 1 segundo para video, < 500 ms para sensores inductivos. Las alertas de conflicto deben tener latencia < 2 segundos. | Para detección de conflictos en tiempo real, la latencia es crítica. El CCO necesita saber de situaciones de riesgo mientras están ocurriendo, no minutos después. |
| **RNF-04** | Resistencia Ambiental | Los sensores de campo deben resistir exposición a lluvia, polvo, radiación UV, vibraciones de tráfico, y potencial vandalismo. Los equipos deben funcionar sin mantenimiento preventivo > 6 meses. | Los equipos están instalados a nivel de calle expuestos a todo. La frecuencia de mantenimiento impacta directamente costos operativos. |
| **RNF-05** | Ciberseguridad y Protección de Datos | Las comunicaciones deben cifrarse (TLS 1.3). Los datos deben anonimizarse antes de transmisión. El acceso a configuración debe requerir autenticación. El sistema debe protegerse contra manipulación de conteos. | Los datos de movilidad pueden ser sensibles. La manipulación de conteos podría afectar decisiones de inversión pública. La protección asegura integridad de datos. |
| **RNF-06** | Eficiencia Energética | El consumo debe ser ≤ 10W promedio por punto de detección para permitir alimentación solar. En modo batería (fallo solar), debe operar ≥ 7 días con capacidad reducida. | Muchos puntos de interés para monitoreo VRU carecen de infraestructura eléctrica cercana. La eficiencia energética habilita despliegue amplio sin cableado costoso. |
| **RNF-07** | Escalabilidad y Estándares Abiertos | La arquitectura debe permitir agregar nuevos puntos de detección sin rediseño. Los datos deb exportarse en formatos estándar (DATEX II, SIRI, GTFS-RT para bicicletas). | La interoperabilidad con otros sistemas de movilidad (apps de ciclismo, planificadores de ruta) maximiza el valor de los datos recolectados. |
