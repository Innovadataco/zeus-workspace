# 6.5 — Proyectos Ágiles en Innovadataco

## Cuándo Usar PM² Agile

| Indicador | ¿Aplica? |
|-----------|----------|
| Desarrollo de producto tecnológico | ✅ Sí |
| MVP (Producto Mínimo Viable) | ✅ Sí |
| Proyecto con requisitos emergentes | ✅ Sí |
| Innovación / Experimentación | ✅ Sí |
| Consultoría con entregables fijos | ❌ Tradicional |
| Proyecto regulatorio estricto | ⚠️ Híbrido |

---

## Configuración Ágil para Innovadataco

### Sprints
- **Duración:** 2 semanas (estándar)
- **Horas por sprint:** ~80 horas (equipo de 2-3 personas)
- **Story points por sprint:** 15-25 (depende de velocity)

### Roles Ágiles
| Rol PM² | Rol Ágil | Persona |
|---------|----------|---------|
| PP | Product Owner | Jelkin |
| DP | Scrum Master + PM | Jelkin / Asignado |
| EDP | Development Team | Equipo técnico + ZEUS |

### Ceremonias
| Ceremonia | Frecuencia | Duración | Participantes |
|-----------|-----------|----------|---------------|
| Sprint Planning | Cada 2 semanas | 2-4 horas | PO, SM, Equipo |
| Daily Stand-up | Diaria (3-4x/semana) | 15 min | Equipo |
| Sprint Review | Cada 2 semanas | 1 hora | Equipo + Stakeholders |
| Retrospective | Cada 2 semanas | 1 hora | Equipo |
| Backlog Refinement | Semanal | 1-2 horas | PO, SM, Equipo |

## Herramientas Ágiles Recomendadas

| Función | Herramienta | Uso |
|---------|-------------|-----|
| Backlog / Kanban | Trello / Notion / Jira | User stories, sprints, tareas |
| Documentación | Notion / Google Docs | DoD, criterios de aceptación |
| Métricas | Spreadsheet / Trello | Velocity, burndown |
| Comunicación | Telegram | Stand-ups asíncronos |
| Demo | Google Meet / Loom | Sprint reviews |
| Versionado | GitHub | Código, releases |

---

## Definition of Done para Innovadataco

```markdown
## Definition of Done

- [ ] Código implementado y funcional
- [ ] Pruebas manuales realizadas (si no hay automatizadas)
- [ ] Revisado por al menos 1 miembro del equipo
- [ ] Documentación técnica actualizada (si aplica)
- [ ] Documentación de usuario actualizada (si aplica)
- [ ] Desplegado en ambiente de pruebas
- [ ] Demo realizado en Sprint Review
- [ ] Aceptado por Product Owner
```

## Integración con PM² Tradicional

```
FASE 1: INICIO (PM² Tradicional)
├── Solicitud de inicio
├── Caso de negocio (light)
└── Acta de constitución

FASE 2: PLANIFICACIÓN (PM² Tradicional + Ágil)
├── Manual del proyecto (light)
├── Roadmap de releases
└── Product Backlog inicial

FASE 3: EJECUCIÓN (PM² Agile)
├── Sprint 1, 2, 3... N
├── Releases cada 2-4 sprints
└── Reviews con stakeholders

FASE 4: CIERRE (PM² Tradicional)
├── Aceptación formal
├── Informe de fin
└── Transición a operaciones
```
