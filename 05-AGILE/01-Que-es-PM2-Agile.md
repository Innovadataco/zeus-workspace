# 5.1 — Qué es PM² Agile

## Definición

**PM² Agile** es una adaptación de la metodología PM² para proyectos que requieren:
- **Entregas frecuentes y tempranas** de valor
- **Requisitos emergentes** que evolucionan durante el proyecto
- **Equipos autoorganizados** y multifuncionales
- **Gestión de cambio como oportunidad**, no como obstáculo

> *"PM²-Ágil no reemplaza ninguna parte de PM². Esta guía amplía PM² proporcionando métodos, herramientas y técnicas específicas que ayudan a los equipos a adoptar y permitir el uso de los principios Ágil y Lean en el contexto de sus proyectos PM²."*

---

## Cuándo Usar PM² Agile

### Indicadores de Aplicabilidad

| Indicador | ¿Usar PM² Agile? |
|-----------|------------------|
| Requisitos poco definidos al inicio | ✅ Sí |
| Tecnología nueva o innovadora | ✅ Sí |
| Necesidad de entregas frecuentes | ✅ Sí |
| Usuario final disponible para feedback | ✅ Sí |
| Equipo pequeño y co-locado | ✅ Sí |
| Requisitos regulatorios estrictos | ⚠️ Mixto |
| Proyecto de infraestructura física | ❌ Tradicional |
| Contrato fijo precio-alcance | ⚠️ Con cuidado |

---

## Diferencias Fundamentales con PM² Tradicional

| Aspecto | PM² Tradicional | PM² Agile |
|---------|-----------------|-----------|
| **Planificación** | Predictiva, detallada al inicio | Adaptativa, evolutiva |
| **Requisitos** | Definidos al inicio | Emergentes, backlog priorizado |
| **Entregables** | Entrega final completa | Incrementos funcionales frecuentes |
| **Documentación** | Extensiva según plantillas | "Justa y necesaria" |
| **Equipo** | Jerárquico, roles definidos | Autoorganizado, multifuncional |
| **Cambio** | Gestión formal, evaluación de impacto | Abrazado como oportunidad |
| **Calidad** | Verificación final | Integrada continuamente |
| **Éxito** | Cumplir plan | Maximizar valor entregado |

---

## El 12º Principio: El Diferenciador Clave

PM² Agile añade un **principio exclusivo** que no existe en el Manifiesto Ágil original:

> *"Las prácticas ágiles deben tener en cuenta las características de la empresa considerando la gobernanza de TI, la arquitectura empresarial y los requisitos de interoperabilidad. Los equipos ágiles deben ser capaces de colaborar eficazmente con equipos y partes interesadas que sigan otros enfoques."*

Este principio refleja la realidad de las grandes organizaciones donde:
- Hay **gobernanza de TI** institucional
- Existen **requisitos de arquitectura empresarial**
- Debe haber **interoperabilidad** entre sistemas
- No todos los equipos pueden ser ágiles simultáneamente

## Estructura de PM² Agile

### Las 4 Fases + Iteraciones

```
┌─────────────┐    ┌─────────────┐    ┌─────────────────────────┐    ┌─────────────┐
│   INICIO    │───→│ PLANIFICACIÓN│───→│      EJECUCIÓN          │───→│   CIERRE    │
│             │    │  (Release    │    │  ┌─────┐┌─────┐┌─────┐  │    │             │
│ Establecer  │    │   Planning)  │    │  │Spr 1││Spr 2││Spr N│  │    │ Formalizar  │
│ fundamentos │    │ Plan inicial │    │  │     ││     ││     │  │    │ cierre      │
│             │    │ de releases  │    │  └─────┘└─────┘└─────┘  │    │             │
└─────────────┘    └─────────────┘    └─────────────────────────┘    └─────────────┘
```

### En la Fase de Ejecución: Sprints

| Elemento | Descripción |
|----------|-------------|
| **Sprint/Iteración** | Ciclo de 1-4 semanas con entregable funcional |
| **Planning** | Planificación del sprint al inicio |
| **Daily Stand-up** | Reunión diaria de 15 minutos |
| **Demo/Review** | Presentación de incrementos al final |
| **Retrospectiva** | Reflexión sobre mejora continua |
| **Backlog** | Lista priorizada de requisitos |

---

## Artefactos Ágiles Adicionales

| Artefacto | Descripción |
|-----------|-------------|
| **Product Backlog** | Lista priorizada de todas las funcionalidades deseadas |
| **Sprint Backlog** | Elementos seleccionados para el sprint actual |
| **Incremento** | Suma de todos los elementos backlog completados |
| **Definition of Done** | Criterios que define cuándo un elemento está terminado |
| **Burndown Chart** | Gráfico de trabajo pendiente vs. tiempo |
| **Velocity** | Velocidad del equipo (puntos por sprint) |

## Roles en PM² Agile

### Adaptación de Roles PM²

| Rol PM² | Rol Ágil | Función |
|---------|----------|---------|
| PP | Product Owner | Define visión, prioriza backlog |
| DP | Scrum Master + PM | Facilita, gestiona proceso |
| EDP | Development Team | Equipo multifuncional autoorganizado |
| RN | Stakeholder Representative | Representa usuarios finales |
| PS | Technical Advisor | Asesora arquitectura técnica |
